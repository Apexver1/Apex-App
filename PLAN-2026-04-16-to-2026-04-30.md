**Created:** 2026-04-16
**Window:** 2026-04-16 → 2026-04-30
**Owner:** Michael Stewart
**Companion docs:** `DATA-STATUS.md` (per-screen real-vs-dummy state), `BACKLOG.md` (deferred work)

---

## North star (the one rule)

**"Login → anything I click → looks great"** for any of 364 D1 schools, for any staff role. Every session ends with that statement true. If we can't keep it true, we ship smaller — never broken.

## Audience reset (this matters)

Apex Intel is a **shared workspace for an entire basketball staff** — head coach, assistant coaches, GM / Director of Basketball Ops, DOBOs, GAs, managers — not a head-coach-only app. Within the 14-day window we are **not** building full role-based access (that's a 1-2 week project tagged in BACKLOG). We *are* tracking who's logged in, attributing actions, and watching for "this works for HC but breaks for GA" tradeoffs.

## Two beta tests in the window

User has scheduled **two real beta tests with industry people** during this 14-day window. These are not internal partner demos — they're real coaches/staff poking at the app and giving feedback. That changes the bar:
- Every session result has to survive a real user clicking around
- We collect feedback after each test and feed it into the plan
- The 14-day plan is revisable after each beta — we don't ride the plan into the ground if a beta tester surfaces a critical gap

Beta dates: TBD (user to add). Plan assumes beta tests fall around Day 5 and Day 12.

## The 10 sessions (ordered, scoped, named)

Sessions are 60–120 min each. User can do 5–8 hrs/day variable; we are NOT booking every day — there is intentional slack for life, bug-fix sessions, beta-feedback work, and re-prioritization.

### Sprint 1 — Foundation (Sessions 1–4)

#### Session 1 — Audit + Sample-badge removal + DATA-STATUS bootstrap (60–90 min)
- Walk every screen for school-switch scenarios (Oregon, Duke, Kansas, Saint Mary's, Howard)
- Remove the Sample/dummy badges shipped in 5f
- Document every broken state, Oregon carryover, ugly empty state in DATA-STATUS.md
- Add a lightweight "current user" concept (user role displayed in drawer; default = Head Coach)
- **Ships:** clean badge-free build + a prioritized bug list + role-aware drawer

#### Session 2 — Universal dummy-data fallback pattern (90 min)
- Replace Oregon-hardcoded seeds (NEXT_GAME, NEWS_SEED, VISITS_SEED, STAFF_SEED, OPS_*) with per-school generators
- Pattern: when live data missing for a school, render dummy that matches the design (no "Sample" tag — invisible to viewer, tracked in DATA-STATUS)
- Apply to Home (NEXT_GAME, NEWS), Visits, Ops first
- **Ships:** every school looks populated on those screens

#### Session 3 — KenPom backend ingestion (90–120 min)
- Verify KenPom API access in `.env` (provider, auth)
- Schema migration on `schools`: `kenpom_rank`, `kenpom_adj_em`, `kenpom_adj_o`, `kenpom_adj_d`, `kenpom_tempo`, `kenpom_season`, `kenpom_last_sync`
- `scripts/kenpom_sync.py` — pull, fuzzy-match (0.94 threshold / 0.04 margin), upsert
- Run sync, spot-check Oregon + 10 schools
- **Ships:** 364 schools have KenPom ranks in DB

#### Session 4 — KenPom frontend surfacing (90 min)
- `schoolKenPomMap` in `loadData()` (mirrors `schoolEspnMap` pattern)
- `kenpomFor(schoolName)` helper
- Wire into: Home Next Up, Roster Pulse, recruiting cards, player modal, scout cards
- **Ships:** KenPom visible everywhere relevant; replaces hardcoded `NEXT_GAME.home.kpom`

**Beta test 1 likely lands around here** (Day 5–6). Plan a feedback-capture session before resuming Sprint 2.

### Sprint 2 — The selling-point screens (Sessions 5–8)

#### Session 5 — Torvik backend + frontend (90–120 min)
- Schema migration for Torvik columns on `schools` (advanced stats: `torvik_adj_oe`, `torvik_adj_de`, etc.)
- Sync script (similar pattern to KenPom)
- Surface in Smart Filter + Apex Scout
- **Ships:** two real data sources backing the selling-point screens

#### Session 6 — Apex Scout polish: the showpiece (90 min)
- This is the screen Michael wants to be best in the app
- Polish copy, polish result cards, ensure every result has rich data
- Add any missing interactions, fix layout issues
- **Ships:** a screen Michael is proud to open first in any demo

#### Session 7 — Smart Filter + Apex Picks preset strip (90 min)
- Now that KenPom + Torvik are wired, build the Apex Picks strip
- Three presets: "Portal PGs, Rebuild Targets" (KenPom bottom-50), "HS 4-Stars, Undercommitted", "Best Value vs NIL Ask"
- Insertion point: `renderFilter` ~line 2643, before first `sectH("Source")` call
- **Ships:** the "killer moment" demo flow

#### Session 8 — NIL Budget per-school + standard defaults (60–90 min)
- Every school gets a default NIL budget on first switch
- Implementation: extend `materialize_school_defaults` RPC OR new RPC `materialize_nil_budget`
- User-adjustable; persists per school
- **Ships:** NIL Budget never empty regardless of school

### Sprint 3 — The long tail + NIL Vault stub (Sessions 9–10)

#### Session 9 — Patrons data for Power 5 + NIL Vault stub (90–120 min)
- Patrons: seed-script for ~65 Power 5 schools (realistic donor names, campaigns, unique per school). Non-Power-5 = polished "Enable Patrons" empty state.
- NIL Vault stub:
  - New drawer entry "NIL Vault" under the player roster section
  - Player profile gains a "Contracts" tab
  - One Oregon player has a real PDF stored (Michael provides) and a static AI-extracted view ("Payment dates: ...", "Deliverables: ...", "Incentive triggers: ...")
  - No upload flow, no live AI, no tracking — just a designed view that demonstrates the concept
  - Real spec lives in BACKLOG as the named project "NIL Contract Vault"
- **Ships:** Patrons never embarrassing; NIL Vault demoable as a vision artifact

#### Session 10 — Beta-test 2 dry run + polish pass (90 min)
- Switch to 5 random D1 schools, walk every tab, fix everything that looks off
- Final polish: spacing, copy, mobile checks
- This is the dress rehearsal for beta test 2
- **Ships:** Michael clicks through the whole app confident in front of any beta tester

### Slack (intentional)

**Days 11–14 are buffer.** Some sessions overrun. Some surface bigger issues. Beta tests will surface things that need same-day fixes. This is not padding — it's the difference between a plan that survives and a plan that collapses.

## Standing session workflow (every session, no exceptions)

1. **Load context** — fetch `CLAUDE.md` + latest handoff + `DATA-STATUS.md` + this plan
2. **State the session target in one sentence.** If I can't write it in one sentence, scope is wrong.
3. **Map the file once** — `grep`-map relevant functions
4. **Plan before code** — numbered steps, risks named, user greenlights
5. **Execute surgically** — contiguous `str_replace` blocks, no find-and-replace
6. **Validate at midpoint and end** — `node --check`
7. **Always-demoable check** — mentally walk login → tab through → any school. If broken, revert, ship smaller.
8. **Deploy** — ship file as `index-{session}.html`, user pushes to main
9. **Update `DATA-STATUS.md`** — log what moved from dummy to real
10. **Write handoff** — `HANDOFF-YYYY-MM-DD-session{N}-{slug}.md`

## What changes about how we work

- **Session naming.** Every session has a number and slug. Handoffs reference both. No more "5f late-night."
